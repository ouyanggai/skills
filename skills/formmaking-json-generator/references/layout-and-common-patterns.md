# 布局与通用能力规律

## 1. 样本基础

- 统计基础：本地上下文中配置的真实 `template_data` 样本目录；没有本地样本时，使用本文件中的经验作为启发式参考
- 规则来源：真实样本 + 当前工作区 FormMaking 源码 + 宿主注册组件源码

## 2. 布局分布

按“表单是否出现过该容器”统计：

- `report`：176 / 178
- `grid`：94 / 178
- `table`：67 / 178
- `inline`：51 / 178
- `subform`：10 / 178

按顶层组合统计，最常见的是：

- 只有 `report`：111
- `grid + report`：39
- `grid + report + table`：9
- `inline + report`：9
- `report + table`：8

结论：

- 宿主里的正式审批表，主骨架默认偏向 `report`
- `grid` 常作为补充分区或响应式包裹层
- `table` 主要承载明细行，不是整张表单的主骨架
- 正式表格型审批表的高频顶层序列，通常是“标题 `text` + 若干个顶层 `report`”

## 3. 布局决策

### `report`

适合：

- 合同、法务、质检、工程、台账类制式审批表
- 有明显行列边界、标题栏、分区栏、合并单元格的表单
- 需要按截图还原视觉结构的场景

不适合：

- 纯粹的移动优先自由排布页面
- 大量响应式换列需求

### `grid`

适合：

- 普通两列/三列信息区
- 作为外层分区骨架包裹标题、卡片、`report`
- 想显式表达“这一块是一个区块”的新生成表单

经验：

- 新生成的复杂表单，推荐每个业务分区先放进一个 `grid`
- 分区内部如果是打印式内容，再嵌 `report`

### `table`

适合：

- 费用明细、税额拆分、主体清单、设备清单、付款节点
- “行结构固定、每行字段少量重复”的数据

注意：

- `table` 是明细容器，不是整张审批表的页面骨架
- 如果需求写了“可新增多行”“删除某行”“按行录入”，优先选 `table`

### `subform`

适合：

- 重复块内部本身又有多字段、甚至有自定义组件
- 不是纯表格行，而是一组可复制业务片段

### `inline`

适合：

- 同一行放 2 到 4 个短字段或只读标签
- 补充局部排版，不建议承载整张表

## 4. 推荐布局套路

### 普通审批表

- 顶部标题：`grid`
- 基础信息：`grid` 或 `report`
- 明细区：`table`
- 附件区：`report` 或 `grid`

### 截图仿制/打印式表单

- 标题区：顶层 `text`
- 每个业务段：优先直接顶层 `report`
- 具体单元格排版：`report`
- 只有当一个业务段内部确实存在多列并排、响应式切块时，再额外引入 `grid`
- 需要增删行的局部：在 `report` 的单元格内放 `table` 或 `subform`

### 合同类复杂表单

- 合同头信息：`report`
- 关联合同/项目/流程：`custom` 组件放在 `report` 单元格或 `grid` 列内
- 主体变更/付款节点：`table`
- 附件与盖章件：单独一个区块，不要塞进明细表里

## 5. 视觉规律

- 表单主体默认应占满中间可用区域，不应因为容器漏宽度而左侧塌缩。
- `report` 是带边框的强容器，顶层表格区通常一层就够；额外多包一层同样带边框的容器，视觉上很容易显得“套娃”。
- 标题通常单独居中，下面的主体表格和标题共享同一视觉中轴。
- 标签单元格一般居中或垂直居中，不会出现标题居中、表格主体偏左的割裂感。
- 截图中的红字大概率是批注，不是最终视觉规范；回填字段、标签文本、说明文本默认都应保持宿主常规色。
- 系统回填值和自动计算值，视觉上要明确“不可编辑”，优先直接走宿主 `disabled` 的灰底灰字，而不是只靠 `readonly`。
- 必填项的红色星号要和宿主现有表单一致，正式表格式场景优先复用 `showRedPot`，不要手写 `*`。

## 6. Word/截图还原补充

- Word/截图来源的表单，布局优先级高于字段语义归并；先还原行列骨架，再选择组件。
- 真实 Word 表格如果有 8/9 个底层网格列，FormMaking 的 `report.headerRow` 也应尽量保留同等逻辑列数，并用 `colspan` 表达合并关系；不要直接压缩成 4/6 列。
- “岗位一、岗位二”“事项一、事项二”这类已经写死编号的分区，默认是固定结构，不是动态重复明细。
- 动态容器的蓝色按钮、编号标签、删除按钮、分页控件会改变制式表单视觉；没有显式增删需求时，避免 `subform + report`。
- Word 里的横向复选/单选选项保持横向控件组，优先 `checkbox`/`radio`；下拉框不是默认替代方案。
- 底部审批意见区优先保持表格轮廓和空白签批区域，是否绑定为可编辑字段需要用户明确要求。
- 色彩迁移遵循“源表有什么才迁移什么”：无底色就白底灰线，有底色才做对应单元格底色；不要为了强调加额外彩色块。

## 7. 反模式

- 顶层 `grid` 只有一个 `span: 24` 列，列里只有一个 `report`
  这通常没有真实布局价值，建议直接把 `report` 放顶层
- 标题单独再包一层单列 `grid`
  正式样本更常见的是直接顶层 `text`
- `report` 漏掉 `options.width = "100%"`
  这是导致轮廓不完整、主体偏左的高风险问题
- 在 `report` 单元格里再放带完整边框的 `report/table`
  除非明确要做嵌套表格，否则容易出现双重轮廓、控件周围边界层级过多
- 把 `subform` 当“排版辅助容器”而不是“重复块容器”
  这会让表单结构复杂化，但不会带来真正的布局收益
- 把源对象信息和当前可编辑信息挤在同一行里
  典型后果是“原值”和“现值”语义混淆；源数据展示区和本次填写区应分开，必要时用“变更前/变更后”明确表达
- 在源数据区重复展示一个字段，同时又在本次填写区提供同名可编辑字段
  这会让用户分不清哪个是只读原值、哪个是本次要填写的值；如果只是默认带入，通常只保留当前可编辑项，并在脚本中从源对象默认赋值
- 每次刷新都把源对象值重新写回可编辑字段
  这会导致用户手动选择后又被覆盖；要用源对象 id 或同步标记判断，只在源对象发生变化时覆盖默认值
- 把 Word 固定编号区块转成 `subform`
  这会产生 Word 原表没有的新增按钮、编号列和内缩轮廓，除非用户明确要动态多条
- 把 Word 的 `□/☑/○/●` 选择项转成 `select`
  这会破坏源表横向排布和一眼可见的选项信息
- 把审批意见区默认全部转成可编辑意见/签字字段
  这会把流程签批信息误当成发起表单字段，除非业务明确需要录入

## 9. 通用组件选择规律

### 人员/部门/公司/岗位

优先 `custome-info-select`

- 这是最稳定的通用选择组件
- 样本节点数：145
- 适合经办人、经办部门、经办公司、合同主体公司、审核人等
- 字段名必须带语义关键字：`userName`、`depName`、`companyName`、`dutyName`
- 组件会额外写入：
  - `<model>__condition`
  - `<model>__formPersonId`

### 公司/部门范围单选或多选

优先 `ltd-or-dep-select`

- 用于集团公司或部门范围选择
- 字段名包含 `singleSelect` 视为单选
- 字段名包含 `mulSelect` 视为多选
- 返回值通常是 JSON 字符串数组

### 单个合同/单个流程对象

优先 `general-list-select-show`

- 样本节点数：31
- 常见于合同、流程、手续文件、单据对象关联
- 查看详情按钮是组件自带能力
- 某些场景需要先通过 `refresh` 注入上下文，例如：
  `this.setData({ contractObj: JSON.stringify({ initiatorCompanyId }) })`
- 如果 `formType == 'contract_receipt_form'`，组件还会检查是否已选收款单位

### 多个流程对象

两类组件不要混用：

- `flow-list-mul-select`
  更偏业务流程多选，差旅/还款类常见，可能要求先选单位
- `general-flow-list-mulSelect`
  更偏通用流程多选，通常先通过 `refresh` 注入：
  `this.setData({ flowObj: JSON.stringify({ flowType: 'xxx' }) })`

### 项目选择

优先 `custome-select-project`

- 样本节点数：16
- 用于集团共享项目或公司项目

## 10. 文件上传规律

### 标准 `fileupload`

适合：

- 普通业务附件
- 合同扫描件、变更附件、请款附件、说明材料

源码事实：

- 默认 `multiple: false`
- 默认 `limit: 100`
- 运行时只接收固定上传参数，不透传文件类型限制
- 实际上传 `<input type=\"file\">` 没有 `accept`

因此：

- 需求写“允许多个”，必须显式设 `multiple = true`
- 需求写“最多 N 个”，必须显式设 `limit = N`
- 需求写“仅 PDF/仅 Excel”，标准 `fileupload` 只能做弱约束：
  - 写 `tip`
  - 写校验脚本
  - 或改用业务专用组件

### 业务专用上传

- `custome-file-import`
  适合“导入模板/导入 Excel 后端解析”
- `custom-upload-excel`
  宿主已注册，但样本很少；只有明确需要时才选

## 11. 生成时的默认策略

- 先选布局，再选字段组件
- 普通正式表单默认不要裸堆字段；至少选 `grid` 或 `report`
- 只要用户给了截图或说“按这个表样做”，优先 `report` 或 `grid + report`
- 但如果截图本身就是标准制式审批表，默认先尝试“顶层 `text` + 顶层 `report`”而不是给每个区块都包 `grid`
- 只要用户说“多行明细/新增一行/删除一行”，优先 `table`
- 只要是人员/部门/公司/岗位选择，先考虑宿主组件，不要先退回普通 `input`
- 交付前至少做一次“反模式回扫”：看有没有漏宽度、单列空包裹、双重边框、标题与主体不共轴
